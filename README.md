# Учёт автомобилей в прокате

Распределённая система управления данными и формирования отчётов.  
Стек: **Python + FastAPI + PostgreSQL + Docker / docker-compose**.

Каждая лабораторная находится в папке `output/labN/` и запускается командой:

```bash
docker-compose up -d --build
```

---

## Лабораторная 1 — Знакомство с Docker

Развёртывание базы данных **PostgreSQL** через docker-compose.  
Создаётся таблица `cars` с полным набором полей согласно схеме.  
Настроен volume для хранения данных и проброс порта `5432` на хост.

---

## Лабораторная 2 — CRUD-сервис

FastAPI-приложение с полным CRUD для сущности `cars`:

- `GET /cars` — список всех автомобилей  
- `GET /cars/{id}` — карточка автомобиля  
- `POST /cars` — принять новый автомобиль  
- `PUT /cars/{id}` — редактировать данные  
- `DELETE /cars/{id}` — удалить запись  
- `POST /cars/{id}/issue` — выдать автомобиль на руки  
- `POST /cars/{id}/write_off` — списать автомобиль  

БД вынесена во внутреннюю Docker-сеть (`internal: true`), снаружи недоступна.  
Swagger UI доступен по адресу `http://localhost:8000/docs`.

---

## Лабораторная 3 — Сервис отчётов

Добавлен отдельный **сервис отчётов** (`report-service`, порт `8001`).  
Эндпоинт `GET /report` возвращает сводку по статусам автомобилей:

```json
{
  "generated_at": "...",
  "summary": { "total_cars": 120, "in_showroom": 58, "issued": 47, "written_off": 15 }
}
```

Оба сервиса (CRUD и отчёты) работают одновременно, подключены к одной БД.

---

## Лабораторная 4 — Изоляция сервиса отчётов

Прямой доступ к `report-service` извне закрыт — сервис убран из публичной сети.  
CRUD-сервис проксирует запросы к отчётам по шаблону `GET /reports/*`, не зная о конкретных методах report-service.

---

## Лабораторная 5 — Файловое хранилище MinIO

Добавлен **MinIO** как объектное хранилище (консоль: `http://localhost:9001`).  
Логика изменена:

1. CRUD-сервис вызывает `POST /report/generate` у report-service.  
2. Report-service формирует отчёт в формате **CSV**, сохраняет в MinIO, возвращает `file_id`.  
3. CRUD-сервис скачивает файл из MinIO и отдаёт клиенту.

Report-service недоступен извне. MinIO доступен для проверки.

---

## Лабораторная 6 — Брокер сообщений RabbitMQ

Добавлен **RabbitMQ** (management UI: `http://localhost:15672`).  
Логика построения отчёта переведена на очереди:

1. CRUD-сервис публикует задание в очередь `report_requests` с `reply_to` и `correlation_id`.  
2. Report-worker (фоновый процесс) читает очередь, формирует CSV, сохраняет в MinIO, публикует `file_id` в callback-очередь.  
3. CRUD-сервис асинхронно ожидает ответа, скачивает файл из MinIO и отдаёт клиенту.

MinIO недоступен извне. RabbitMQ доступен для проверки.

### Выводы

- **Чтение** масштабируется лучше всего — при 50 потоках P95 остаётся в пределах 651 мс.  
- **Запись** показывает меньшую задержку, чем смешанный сценарий, благодаря отсутствию конкуренции за чтение.  
- **Смешанный сценарий** демонстрирует резкий рост задержки при 50 потоках (P95 > 5 с) из-за конкуренции за ресурсы БД.  
- **Запись + отчёт** через RabbitMQ имеет предсказуемое время ответа — P95 не превышает 1.4 с даже при 50 потоках.
